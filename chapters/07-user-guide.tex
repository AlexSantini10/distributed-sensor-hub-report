\section{User Guide}\label{user-guide}

This section describes operational system usage from the perspective of a technical user who wants to start the cluster, observe state convergence, and inspect introspection surfaces. Since the project does not expose external write capabilities, the main interaction consists of starting nodes, waiting for distributed-system evolution, and analyzing behavior through HTTP APIs and the web dashboard.

\subsection{System startup}

The most effective setup for demonstration is to start an already prepared Docker topology. The recommended scenario is the six-node one:
\begin{verbatim}
docker compose -f docker/docker-compose-6-nodes.yml up --build -d
\end{verbatim}

After startup, it is advisable to wait a few seconds so that nodes can complete bootstrap, peer-list exchange, initial heartbeats, and initial state replication. The expected effect is the progressive appearance of a shared membership view and sensor records replicated across nodes.

To stop the cluster:
\begin{verbatim}
docker compose -f docker/docker-compose-6-nodes.yml down
\end{verbatim}

\subsection{Direct API inspection}

Each node exposes a read-only web API on the port defined by \texttt{WEB\_API\_PORT}. The most useful endpoints during operation are:
\begin{itemize}
  \item \texttt{/api/state}: snapshot of replicated sensor state;
  \item \texttt{/api/updates}: stream or view of recent updates;
  \item \texttt{/api/membership}: local membership view;
  \item \texttt{/api/topology}: representation of known topology;
  \item \texttt{/api/introspection}: aggregated snapshot according to the \texttt{introspection/v1} contract;
  \item \texttt{/api/introspection/events}: recent control-plane events;
  \item \texttt{/api/introspection/metrics}: counters and indicators related to replication.
\end{itemize}

For example, in a base topology, \texttt{node-1} is normally reachable at \texttt{http://localhost:10000}. Querying the aggregated \texttt{/api/introspection} endpoint provides a useful view to verify:
\begin{itemize}
  \item which peers are currently considered \texttt{alive}, \texttt{suspected}, or \texttt{dead};
  \item which sensor records are known to the queried node;
  \item how many replication rounds and gossip messages have been observed;
  \item whether the delta buffer is sufficient or produces \texttt{DELTA\_UNAVAILABLE}.
\end{itemize}

\subsection{Using the web dashboard}

For more convenient observation, a static dashboard is available in the \texttt{web/} directory. It requires no additional backend and consumes only the \texttt{/api/introspection} endpoint. To start it:
\begin{verbatim}
cd web
python -m http.server 8080
\end{verbatim}

Then open \texttt{http://localhost:8080} in the browser and set the \emph{API Base URL} to the HTTP endpoint of one node, for example \texttt{http://localhost:10000}.

The dashboard provides the main analysis views:
\begin{itemize}
  \item graph of observed topology;
  \item peer status and related liveness semantics;
  \item table of replicated sensors with source and timestamp;
  \item timeline of recent events;
  \item synthetic metrics strip for gossip, full sync, and delta replication.
\end{itemize}

In a stable cluster, the expected outcome is a topologically coherent view, a majority of peers in \texttt{alive} state, and relatively recent sensor timestamps for high-frequency sources.

\subsection{Interpreting key observable signals}

During operation, it is useful to interpret several operational indicators:
\begin{itemize}
  \item if many \texttt{sensor\_update\_received} events appear with \texttt{applied:false}, the node is receiving updates already superseded by the LWW criterion;
  \item if the \texttt{get\_delta\_unavailable\_total} counter increases, the incremental-delta buffer may be too short for current load or reconnection times;
  \item if a peer frequently transitions from \texttt{alive} to \texttt{suspected}, the simulated network or heartbeat interval may be introducing transients visible to the failure detector;
  \item if sensor timestamps appear too old in the UI, it is advisable to reduce pull pressure or increase \texttt{REPLICATION\_DELTA\_MAXLEN}.
\end{itemize}

The dashboard is therefore not only a demonstration tool, but also an important component for experimental diagnosis of distributed behavior.

\subsection{Recommended usage scenarios}

From a didactic and demonstration standpoint, the most significant scenarios are:
\begin{itemize}
  \item \textbf{convergence under normal conditions}: start the cluster and observe progressive state alignment;
  \item \textbf{crash and recovery}: temporarily stop one node and verify catch-up through deltas or full sync;
  \item \textbf{degraded network}: use delays and packet loss configured in Compose to observe effects on heartbeats and data freshness;
  \item \textbf{observable scalability}: compare 2-node, 6-node, and 12-node topologies through the same UI.
\end{itemize}

For repeated fault-injection scenarios, the script \texttt{manual\_tests/compose\_chaos.py} is also available and useful to introduce controlled stop/restart cycles of individual services while the cluster is running.

\subsection{Using an image published through release}

If building the Docker image locally is not desired, it is possible to use a release generated by the continuous-delivery pipeline directly. However, a release is not created for every commit: it is produced when a version tag is assigned to a commit, for example \texttt{v1.0.0}, associated with the node image uploaded to GitHub Container Registry.

The operational flow is as follows:
\begin{enumerate}
  \item create the version tag locally and publish it to the remote repository;
  \item open the repository \emph{Releases} section;
  \item select the release corresponding to the desired commit;
  \item download attached assets, especially the example \texttt{.env} file and minimal \texttt{docker-compose};
  \item adapt the main parameters of the \texttt{.env} file;
  \item start the container through Compose or use the \texttt{docker run} command reported in release notes.
\end{enumerate}

An example of release publication is:
\begin{verbatim}
git tag v1.0.0
git push origin v1.0.0
\end{verbatim}

A typical execution example using attached assets is:
\begin{verbatim}
cp docker-node.env.example docker-node.env
docker compose -f docker-compose.release.yml up -d
\end{verbatim}

Alternatively, direct startup by digest is also available:
\begin{verbatim}
docker pull ghcr.io/<owner>/<repo>@sha256:<digest>
docker run --rm -p 9000:9000 -p 10000:10000 \
  -e NODE_ID=node-1 \
  -e HOST=0.0.0.0 \
  -e PORT=9000 \
  -e WEB_API_PORT=10000 \
  -e BOOTSTRAP_PEERS= \
  ghcr.io/<owner>/<repo>@sha256:<digest>
\end{verbatim}

This mode is useful when rapidly running a node consistent with a specific commit on the main branch, while preserving environment reproducibility through an immutable reference to the image digest.
