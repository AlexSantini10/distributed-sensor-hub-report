\section{Concept}\label{concept}

\textbf{Product type.}
The software is a distributed experimental platform composed of autonomous peer-to-peer nodes and an optional web monitoring interface.
Concretely, it combines:
\begin{itemize}
  \item a set of autonomous nodes that collect and replicate sensor data without a central coordinator;
  \item an optional read-only web/introspection layer for monitoring and runtime observability.
\end{itemize}


\textbf{Use case description.}
The system models a smart-city scenario in which multiple nodes ingest local sensor readings, maintain a local view of shared data, and exchange updates across the peer network. The project is intentionally framed as a controlled distributed-systems laboratory rather than a production-ready urban platform, so that the core distributed properties can be observed and studied directly.

The platform supports two main interaction profiles:
\begin{itemize}
  \item \emph{observational interaction}: users inspect sensor values, node health, and topology evolution;
  \item \emph{operational interaction}: users start experiments, configure nodes, and observe behavior under faults.
\end{itemize}

Users interact through standard HTTP APIs and, optionally, through a browser-based monitoring UI. Data handled by the system is technical runtime information (sensor values, node metadata, and observability data), not personal user information.

Interaction channels and devices:
\begin{itemize}
  \item HTTP API for programmatic interaction, monitoring, and integration with external tools;
  \item desktop/laptop web browser for optional dashboard-based monitoring.
\end{itemize}

\textbf{Why distribution is needed.}
Distribution is a core requirement, not an implementation detail:
\begin{itemize}
  \item sensing points are geographically and logically distributed;
  \item no single coordinator should be required for normal operation;
  \item communication delays, crashes, and temporary partitions must be tolerated;
  \item node-local views must eventually realign after faults are resolved.
\end{itemize}

The detailed behavioral requirements, protocol constraints, and acceptance criteria are specified in \cref{requirements}.
