\section{Implementation}\label{implementation}

This chapter reports the main technology-dependent choices used to realize the design in \cref{sec:design}.

\subsection{Protocols and Data Representation}
\label{implementation-protocols}

The implementation uses two communication planes with different responsibilities.

\textbf{Node-to-node communication.}
Peer communication is implemented over \textbf{TCP}, with a custom application protocol and JSON payloads. TCP is used for:
\begin{itemize}
  \item bootstrap and peer discovery (\texttt{JOIN\_REQUEST}, \texttt{PEER\_LIST});
  \item heartbeat liveness traffic (\texttt{PING}, \texttt{PONG});
  \item membership/topology/state view convergence (\texttt{GOSSIP\_STATE}, distinct from heartbeat and discovery flows);
  \item replicated sensor updates (\texttt{SENSOR\_UPDATE});
  \item incremental and full catch-up (\texttt{GET\_DELTA}, \texttt{DELTA\_UNAVAILABLE}, \texttt{FULL\_SYNC\_REQUEST}, \texttt{FULL\_SYNC\_RESPONSE}).
\end{itemize}

Replication is implemented as a push-pull runtime: nodes periodically push local deltas and periodically request missing deltas from sampled alive peers, with fallback to full sync when incremental history is unavailable.

\textbf{Observer communication.}
External users/tools interact through a \textbf{HTTP API} implemented with \texttt{ThreadingHTTPServer}. The HTTP surface is intentionally observation-oriented: it exposes read endpoints (e.g., state, updates, membership, topology, introspection) and also serves static dashboard assets, while remaining separate from peer coordination logic.

\textbf{In-transit representation.}
Protocol messages and HTTP responses are represented as \textbf{JSON} (UTF-8), chosen for inspectability and interoperability with browser/Python tooling.

\subsection{State Management and Storage Choices}
\label{implementation-storage}

The implementation uses \textbf{in-memory state only}; no external database is used. Each node maintains:
\begin{itemize}
  \item current winning sensor records;
  \item bounded ordered replication deltas and per-origin sequence tracking;
  \item peer membership/liveness state;
  \item topology state;
  \item introspection snapshots (events/metrics).
\end{itemize}

Sensor convergence is implemented in application logic via deterministic LWW merge semantics on \texttt{(timestamp, origin)}. Membership and liveness are maintained as local views updated by protocol handlers and periodic runtime components.

\textbf{Membership and topology runtime model.}
Membership is maintained per node in a local \texttt{PeerTable}, updated by discovery traffic (\texttt{JOIN\_REQUEST}, \texttt{PEER\_LIST}), heartbeat evidence (\texttt{PING}/\texttt{PONG}), and gossip exchanges. Peer status is not binary up/down: it is evaluated as \texttt{alive}/\texttt{suspect}/\texttt{dead} from time-based evidence and configurable phi thresholds (e.g., \texttt{phi\_threshold\_suspect}, \texttt{phi\_threshold\_dead}).

Membership updates use deterministic timestamp-based merge logic for liveness/status information, so node-local views still converge when updates are received in different orders. \texttt{GOSSIP\_STATE} carries not only peer presence but also metadata useful for convergence of cluster-view, observed topology state, and node-reported state summaries used by the introspection layer.

Topology is represented as explicit runtime state in a dedicated in-memory component (\texttt{TopologyStateStore}), updated by gossip/runtime flows and exposed as a read-only snapshot. Link-selection behavior is policy-driven (currently \texttt{full\_mesh}), configured through \texttt{TOPOLOGY\_POLICY}, so topology behavior can evolve without changing the peer protocol.

For observability, membership and topology snapshots are exposed through HTTP endpoints (e.g., \texttt{/api/membership}, \texttt{/api/topology}). Unlike sensor-state LWW records, membership/topology follow dedicated liveness/operational merge rules and are not treated as ordinary sensor-value entities.

\subsection{Authentication and Authorization}
\label{implementation-auth}

No explicit authentication/authorization is implemented in the current version (scope choice for controlled lab/development setups):
\begin{itemize}
  \item peer channels are not authenticated;
  \item HTTP endpoints do not require tokens/sessions/login;
  \item no RBAC/ABAC layer is implemented.
\end{itemize}

There is currently no TLS/mTLS protection on transport channels. Therefore, reachability to an exposed node interface implies access to that interface. Security hardening is future work.

\subsection{Technological details}\label{technological-details}

\textbf{Core technologies used}:
\begin{itemize}
  \item \textbf{Python} runtime and standard library (\texttt{socket}, \texttt{threading}, \texttt{ThreadingHTTPServer}, \texttt{json}, \texttt{logging});
  \item \textbf{thread-based concurrency} (server threads, worker pools, periodic background threads, queue-based local sensor ingestion);
  \item \textbf{environment-based configuration} via \texttt{.env} and \texttt{python-dotenv};
  \item \textbf{Docker}/\textbf{Docker Compose} for reproducible multi-node execution;
  \item \textbf{pytest} and \textbf{pyright} for testing and static analysis.
\end{itemize}

\textbf{Key implementation mechanisms}:
\begin{itemize}
  \item explicit TCP application framing with server-side frame-size validation;
  \item inbound backpressure: the node accepts only a bounded number of connections/workers and rejects excess load;
  \item resilient outbound channels: per-peer message queues with retry/backoff and automatic reconnect;
  \item push-pull replication tuning via configurable probabilistic fanout/cadence parameters (push ratio, pull ratio, pull cadence);
  \item bounded incremental delta log (\texttt{replication\_delta\_maxlen}) to control memory usage and trigger full-sync fallback when history is insufficient;
  \item per-origin cursor tracking (\texttt{latest\_seq\_by\_origin}) to resume incremental recovery from correct sequence points;
  \item configurable runtime fault injection (delay/jitter/loss) for robustness testing without external fault-injection tooling.
\end{itemize}

Overall, the stack is intentionally lightweight and explicit, prioritizing observability, reproducibility, and deterministic behavior over framework-heavy abstractions.

In discursive terms, these implementation choices directly realize the requirements discussed in \cref{requirements}, especially the sections on distributed behavior, resilience under failures, and operational observability.
