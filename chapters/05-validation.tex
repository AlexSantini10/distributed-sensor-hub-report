\section{Validation}\label{validation}

Project validation was designed across multiple layers to verify both the functional correctness of individual modules and the emergent behavior of the distributed system under replication, failures, and reconnection. The adopted strategy combines unit tests, integration tests, smoke tests in a containerized environment, and manual observation scenarios, with the goal of covering the entire node lifecycle: startup, bootstrap, state propagation, failure detection, recovery, and introspection.

\subsection{Automatic Testing}\label{automatic-testing}

\textbf{Unit tests.}
The foundation of the validation plan is a unit test suite developed with \texttt{pytest}. Unit tests verify, in isolation, the responsibilities of the system's main modules:
\begin{itemize}
  \item \texttt{tests/protocol/}: correctness of serialization, deserialization, validation, and dispatch of application-protocol messages;
  \item \texttt{tests/networking/}: TCP framing, client/server connection handling, operational limits, and transport robustness;
  \item \texttt{tests/state/}: deterministic application of the Last-Writer-Wins policy and maintenance of incremental deltas;
  \item \texttt{tests/membership/} and \texttt{tests/fd/}: membership-view updates, heartbeat handling, and \emph{phi-accrual} estimation for peer classification;
  \item \texttt{tests/sensors/}: behavior of simulated sensor generators and the related lifecycle manager;
  \item \texttt{tests/webapi/} and \texttt{tests/introspection/}: consistency of read-only HTTP endpoints and of the introspection contract exposed to clients.
\end{itemize}

The rationale for this suite is twofold: on the one hand, to prevent regressions in components with local semantics; on the other hand, to make explicit the invariants that the system must preserve regardless of distributed topology. At the time of local repository verification, running \texttt{python -m pytest -m "not integration" -q} produced \texttt{180 passed, 9 deselected}, confirming the stability of the non-integrated portion of the suite.

\textbf{In-process integration tests.}
A second validation layer exercises interaction among multiple real nodes executed within the same test process or in processes coordinated by dedicated harnesses. In particular:
\begin{itemize}
  \item \texttt{tests/integration/test\_deterministic\_state\_convergence.py} verifies replicated-state convergence in a six-node cluster, checking that final LWW winners match expected outcomes;
  \item \texttt{tests/integration/test\_cluster\_harness.py} validates the correct behavior of the supporting infrastructure for reproducible distributed scenarios;
  \item \texttt{tests/integration/test\_finite\_test\_sensor.py} uses deterministic sources to make update sequences observable and comparable across nodes.
\end{itemize}

These tests cover properties that cannot be inferred from local correctness alone: incremental propagation, relative ordering of updates, initial bootstrap, and eventual convergence after multiple message exchanges.

\textbf{Docker-based integration tests and production-like scenarios.}
The third validation layer uses Docker Compose topologies to exercise the software in an environment closer to real deployment. This category includes:
\begin{itemize}
  \item \texttt{tests/integration/test\_docker\_deterministic\_state\_convergence.py}, which replicates convergence verification on containerized nodes;
  \item \texttt{tests/integration/test\_docker\_crash\_restart\_recovery.py}, which evaluates recovery after node stop and restart;
  \item \texttt{tests/integration/test\_docker\_partition\_reconciliation.py}, which verifies state reconciliation after a connectivity discontinuity;
  \item \texttt{tests/integration/verify\_cluster.py}, used as a smoke test to check correct cluster startup and reachability of primary endpoints.
\end{itemize}

The main advantage of this solution is that the same deployment setup used for demonstrations is reused as test infrastructure. This reduces the distance between development, verification, and presentation environments, improving the reproducibility of experiments.

\textbf{Execution automation.}
Automated test execution is also supported by CI workflows defined in \texttt{.github/workflows/unit-tests.yml} and \texttt{.github/workflows/integration-tests.yml}. The unit-test pipeline installs dependencies and runs \texttt{pytest} while excluding integration tests, whereas the integration pipeline runs both deterministic convergence checks and a smoke test on Docker clusters. This automation helps detect regressions in both local logic and the most relevant distributed scenarios.

\textbf{How to run the tests.}
The main execution modes, already documented in the repository, are the following:
\begin{itemize}
  \item unit and modular tests: \texttt{pytest --maxfail=1} or \texttt{python -m pytest -v -m "not integration"};
  \item protocol tests: \texttt{pytest -m protocol};
  \item targeted verification of LWW convergence: \texttt{python -m pytest tests/integration/test\_deterministic\_state\_convergence.py -q -s};
  \item Docker smoke test: start the Compose topology, then run \texttt{python tests/integration/verify\_cluster.py --timeout 60 --interval 2}.
\end{itemize}

\textbf{Coverage with respect to project goals.}
Overall, automated tests cover the most important points identified in the project requirements:
\begin{itemize}
  \item correctness of the communication protocol and its validations;
  \item deterministic convergence of replicated state;
  \item propagation of membership and liveness information;
  \item tolerance to crashes, restarts, and temporary partitions;
  \item external verifiability through HTTP APIs and introspection.
\end{itemize}

More explicitly, the main test groups map to requirements as follows:
\begin{description}
  \item[\texttt{tests/state/test\_lww.py}] verifies Last-Writer-Wins merge behavior, stale-update rejection, tie breaking, full-state merge, partition convergence, delta buffers, and cursors. It covers FR3, FR4, FR7, FR9, FR11, NFR2, and NFR3.
  \item[\texttt{tests/fd/} and \texttt{tests/membership/}] verify phi estimation, peer-table updates, heartbeat transitions, suspected/dead/alive recovery, gossip status merging, and membership snapshots. They cover FR5, FR6, FR10, and NFR4.
  \item[\texttt{tests/protocol/test\_full\_sync\_handlers.py}] verifies \texttt{GET\_DELTA}, \texttt{DELTA\_UNAVAILABLE}, full-sync requests, full-sync responses, and recovery fallback. It covers FR7, FR9, FR11, and NFR2.
  \item[\texttt{tests/protocol/test\_gossip\_state\_handlers.py}] verifies membership gossip, stale status rejection, delayed convergence after partitions, and topology merge. It covers FR5, FR6, NFR1, NFR3, and NFR4.
  \item[\texttt{tests/runtime/test\_sensor\_update\_publisher.py}] verifies push-pull fanout, alive-peer targeting, and pull tracking. It covers FR3, FR9, NFR2, and NFR7.
  \item[\texttt{tests/webapi/} and \texttt{tests/introspection/}] verify read-only HTTP endpoints and aggregate introspection snapshots. They cover FR8, FR10, FR12, and NFR4.
  \item[\texttt{tests/sensors/}] verifies simulated sensor providers and sensor-manager configuration. It covers FR2 and NFR6.
  \item[\texttt{tests/networking/}] verifies TCP framing, connection handling, concurrency, slow clients, bursts, limits, and shutdown. It covers FR1, FR3, NFR1, and NFR7.
  \item[\texttt{tests/topology/}] verifies topology-state merge and full-mesh policy resolution. It covers FR5, FR8, NFR7, and IR3.
  \item[\texttt{tests/integration/test\_deterministic\_state\_convergence.py}] verifies that six real in-process nodes converge to deterministic expected winners. It covers FR1, FR2, FR3, FR4, FR9, NFR1, NFR2, NFR7, and IR2.
  \item[\texttt{tests/integration/test\_cluster\_harness.py}] verifies that the reusable cluster harness starts nodes and exposes state and membership. It covers FR1, FR5, FR8, IR1, and IR2.
  \item[\texttt{tests/integration/test\_docker\_deterministic\_state\_convergence.py}] verifies convergence in container networking. It covers FR1, FR2, FR3, FR4, FR9, NFR2, NFR5, NFR7, IR1, and IR2.
  \item[\texttt{tests/integration/test\_docker\_crash\_restart\_recovery.py}] verifies crash/restart recovery, state reconciliation, and membership recovery. It covers FR6, FR7, FR10, FR11, NFR3, NFR4, and NFR5.
  \item[\texttt{tests/integration/test\_docker\_partition\_reconciliation.py}] verifies temporary network partition, divergence, healing, and reconciliation. It covers FR3, FR4, FR7, FR9, FR11, NFR2, and NFR3.
  \item[\texttt{tests/integration/verify\_cluster.py}] performs a Docker smoke check that each node exposes replicated data from expected origins. It covers FR1, FR3, FR8, NFR5, IR1, and IR2.
\end{description}

The rationale of the main distributed tests is also requirement-driven. Deterministic convergence tests prove that autonomous nodes exchange finite sensor streams and converge to the same Last-Writer-Wins result, intercepting silent replication failures, incorrect fanout, or non-deterministic merge behavior. Docker convergence repeats the same idea in container networking, where DNS names, exposed ports, startup timing, and network isolation are closer to deployment conditions. Crash/restart recovery validates that a restarted node, after losing volatile in-memory state, can catch up through delta exchange or full synchronization, while surviving peers continue operating. Partition reconciliation checks that the system can remain locally available during a split, accept temporary divergence, and converge again after communication is restored. Full-sync handler tests specifically protect against the risk that stale cursors make recovery impossible, while web API and introspection tests protect the dashboard and external tools from unstable or incomplete observation contracts.

\subsection{Acceptance test}\label{acceptance-test}

Alongside automated validation, manual acceptance tests were also executed, primarily focused on the system's observable and experimental aspects. These tests were performed through:
\begin{itemize}
  \item startup of 2-node, 6-node, and 12-node Docker clusters;
  \item direct inspection of \texttt{/api/state}, \texttt{/api/membership}, \texttt{/api/introspection}, and related derived views;
  \item use of the static dashboard in \texttt{web/} to observe topology, recent events, replication metrics, and peer status;
  \item execution of \texttt{manual\_tests/compose\_chaos.py} to selectively stop and restart services during cluster execution.
\end{itemize}

These manual tests were intended to confirm properties that are difficult to capture fully through automated assertions alone: readability of exported state, quality of observability, temporal consistency of dashboards, and perceivable system behavior during reconfiguration and network transients.

The manual component does not replace the automated suite, but complements it. In particular, it proved useful when acceptance criteria were also qualitative, for instance in evaluating the clarity of information shown to users or in visually diagnosing phenomena such as pull storms, accumulated delays, or rapid membership state transitions.
