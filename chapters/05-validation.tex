\section{Validation}\label{validation}

Validation follows a layered approach: unit tests for local correctness, integration tests for multi-node behavior, Docker-based tests for production-like conditions, and manual acceptance checks for observability quality.

\subsection{Automatic Testing}\label{automatic-testing}

\textbf{Unit and modular tests.}
Unit tests are implemented with \texttt{pytest} and validate component-level responsibilities:
\begin{itemize}
  \item protocol parsing/validation/dispatch (\path{tests/protocol/*});
  \item networking robustness and limits (\path{tests/networking/*});
  \item LWW state merge and delta handling (\path{tests/state/*});
  \item membership and phi-accrual failure detection (\path{tests/membership/*}, \path{tests/fd/*});
  \item topology policies and in-memory topology state (\path{tests/topology/*});
  \item sensor providers (\path{tests/sensors/*});
  \item read-only observation endpoints (\path{tests/webapi/*}, \path{tests/introspection/*}).
\end{itemize}

Rationale: prevent local regressions and enforce core invariants independently of topology.

\textbf{Harness (in-process integration).}
In-process integration tests exercise interaction among real nodes coordinated by local harnesses:
\begin{itemize}
  \item harness component: \path{tests/integration/cluster_harness.py};
  \item harness validation: \path{tests/integration/test_cluster_harness.py};
  \item deterministic convergence scenario: \path{tests/integration/test_deterministic_state_convergence.py};
  \item finite deterministic sensor flow scenario: \path{tests/integration/test_finite_test_sensor.py}.
\end{itemize}

These tests target properties that unit tests alone cannot prove: bootstrap behavior, propagation ordering effects, and eventual convergence.

\textbf{Docker (production-like integration).}
Docker-based integration tests run in containerized Compose topologies:
\begin{itemize}
  \item harness component: \path{tests/integration/docker_cluster_harness.py};
  \item Docker availability guards: \path{tests/integration/docker_requirements.py};
  \item \path{tests/integration/test_docker_deterministic_state_convergence.py}: verifies that, with deterministic workload in a containerized multi-node cluster, all nodes converge to the same final state view.
  \item \path{tests/integration/test_docker_partition_reconciliation.py}: verifies that after a temporary network partition, nodes reconcile missing updates (delta/full-sync fallback) and converge again once connectivity is restored.
  \item \path{tests/integration/test_docker_crash_restart_recovery.py}: verifies that a restarted node re-joins the cluster, recovers current state from peers, and reaches convergence again without manual intervention.
\end{itemize}

Rationale: reuse the same deployment style used in demonstrations, reducing environment drift.

The Docker integration scenarios use dedicated Compose configurations, in particular \path{docker/docker-compose-integration-tests.yml} and \path{docker/docker-compose-integration-deterministic.yml}.

\textbf{Execution automation.}
Automated execution is provided by CI workflows (\path{.github/workflows/unit-tests.yml}, \path{.github/workflows/integration-tests.yml}). In the current CI configuration, \path{.github/workflows/unit-tests.yml} runs unit/modular suites, while \path{.github/workflows/integration-tests.yml} runs the deterministic in-process integration check (\path{tests/integration/test_deterministic_state_convergence.py}).

\textbf{How to run the tests.}
Main execution commands:
\begin{itemize}
  \item unit/modular tests:
\begin{verbatim}
python -m pytest -v -m "not integration"
\end{verbatim}
  \item targeted multi-node convergence:
\begin{verbatim}
python -m pytest tests/integration/test_deterministic_state_convergence.py -q -s
\end{verbatim}
  \item Docker integration validation (examples):
\begin{verbatim}
python -m pytest tests/integration/test_docker_deterministic_state_convergence.py -q -s
python -m pytest tests/integration/test_docker_partition_reconciliation.py -q -s
python -m pytest tests/integration/test_docker_crash_restart_recovery.py -q -s
\end{verbatim}
\end{itemize}

\textbf{Coverage and rationale.}
The three-layer strategy (unit, harness, Docker) covers the central project goals:
\begin{itemize}
  \item protocol correctness and validation;
  \item deterministic replicated-state convergence;
  \item membership/liveness dissemination;
  \item crash, restart, and partition tolerance;
  \item read-only API and introspection correctness.
\end{itemize}

In summary, validation provides layered coverage (unit + integration) across protocol, networking, state convergence, membership/topology, and fault resilience.
Requirement linkage is explicit in test modules and covers the distributed-behavior, resilience, and observability targets described in \cref{requirements}.

Out of scope for this validation: formal verification, Byzantine tolerance, security hardening guarantees, and durable recovery after total cluster loss.

\subsection{Acceptance test}\label{acceptance-test}

Manual acceptance tests complement automation and focus on aspects that are difficult to validate through assertions alone:
\begin{itemize}
  \item readability and coherence of observation outputs (\texttt{/api/state}, \texttt{/api/membership}, \texttt{/api/introspection});
  \item dashboard consistency during runtime evolution (\texttt{/ui});
  \item perceivable behavior during fault/recovery scenarios (including controlled stop/restart via \path{manual_tests/compose_chaos.py}).
\end{itemize}

Manual testing remains useful because these checks are partly qualitative (human-facing observability and diagnostic clarity), while automated tests remain the primary correctness mechanism.
